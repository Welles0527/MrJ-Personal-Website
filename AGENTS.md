# 项目规则

## 项目默认原则

- 以下情况先说明影响、涉及文件、验证方式并等我确认：
  - 新增依赖、修改 `package.json`、部署配置
  - 接入登录、支付、数据库或第三方 API
  - 修改数据结构、权限、路由
  - 一次涉及超过 3 个核心文件
  - 预计需要多轮排查或长时间构建
- 如果当前 Codex 不支持斜杠命令，按命令后的中文目标执行；不要因为命令不可用而跳过对应步骤。
- 不要每次完整全量校验：只校验改动文件，而非全站代码；只修改用户本次明确要求的问题。
- 先定位相关文件，再做最小改动。
- 不要全仓库扫描，除非定位失败。
- 不要重构、升级依赖、改目录结构或重写整个文件，除非用户明确要求。
- 保持现有布局、路由、数据结构、解锁逻辑和视觉方向不变。

## UI 修改规则

- 修改 UI 后必须运行本地页面并检查效果。
- 桌面端至少检查 `1440x900`。
- 小屏端至少检查 `390x844`。
- 弹框、按钮、浮层必须确认没有溢出、遮挡、错位。
- 不允许只根据 CSS 直觉说完成。

## 功能修改规则

- 修改按钮、表单、登录、权限、提交、跳转、数据保存后，必须手动或自动验证功能链路。
- 如果无法验证，必须明确说明“未验证”。

## 任务拆分规则

- 拆分任务：单独 lint、单独 build、单独页面验收，分开执行以减少单次上下文长度。
- 设定熔断阈值：单次构建或校验超过 20 步直接终止，避免无限迭代堆高成本。

## 代码修改与验证规则

以下命令需在用户以对应命令或明确中文目标发起后执行；若当前 Codex 不支持斜杠命令，按命令后的中文目标执行。

- `/code`：用户以此发起小改动、明确 bug、单个按钮或单个样式问题时，直接进入实现；不必每次都写大规格，节省 token。
- `/lint`：检查语法、格式、潜在错误。每完成一小段修改后、提交前执行；这是机器确定性检查，比 AI 自我感觉可靠。
- `/run`、本地预览、截图：运行真实页面。所有 UI 修改后执行，用于确认网页视觉效果及弹窗是否错位；lint 无法覆盖这些问题。
