# 开发路线图：工程化 Vibe Coding 的 9 个阶段

> 这份文档是**你的施工蓝图**。它告诉你：每个阶段做什么、为什么做、做完该拿到什么、怎么检查合格。
> 怎么跟 Codex 开口要这些成果，看 [`02-Codex沟通手册.md`](./02-Codex沟通手册.md)。
> 快速入口看 [`00-启动清单.md`](./00-启动清单.md)。

---

## 0. 怎么用这份文档

- **按顺序读，按顺序做。** 阶段 0 → 9，前面的阶段是后面的地基，不要跳。
- **每个阶段四件套**：`做什么` / `为什么` / `交付物` / `验收标准`。你只需要盯着"交付物"和"验收标准"——满足就进下一阶段，不满足就让 Codex 改。
- **一个阶段一次 Git 提交。** 做完、验收通过、立刻让 Codex 提交一个版本（"快照"）。这样永远能回退。
- **遇到术语不懂**：直接问 Codex"用大白话解释一下 XX 是什么"，或者查文末的速查卡。

## 1. 核心理念：工程化 vs 业余 vibe coding

业余 vibe coding 是这样翻车的：

> ❌ "帮我做一个个人网站，要好看。" → AI 一次生成一大坨代码 → 你看不懂、改不动、手机上歪了、想加个功能全崩了、最后推倒重来。

工程化 vibe coding 的 5 条原则，正好对症：

| # | 原则 | 一句话解释 | 反面教材 |
|---|------|-----------|----------|
| 1 | **先计划后动手** | 写代码前，先把"要什么"写成文档（PRD） | 直接让 AI 开写 |
| 2 | **用文件存上下文** | 重要决策、架构、进度都写成文件存项目里，不靠聊天记录记 | 决策只在脑子里/聊天里 |
| 3 | **小步快跑** | 一次只让 Codex 做一小块（一个页面、一个功能），做完验收再做下一块 | 一次让它做完整站 |
| 4 | **每步可验证** | 每个阶段都要能"跑起来看到东西"，能看就不是空中楼阁 | 一堆代码但跑不起来 |
| 5 | **勤做版本管理** | 每个阶段用 Git 存一个版本，改坏了能回到上一版 | 改坏了找不到原样 |

> 💡 一句话总结：**你想清楚"要什么"，Codex 负责"怎么写"，文档和 Git 把过程钉死。**
> 这就是你一个非程序员，也能做出"架构清晰、可维护"网站的底气。

---

## 2. 开发流程总览（9 阶段全景）

```
阶段0 初始化 → 阶段1 PRD → 阶段2 选型 → 阶段3 找参考
   → 阶段4 架构 → 阶段5 分页面 → 阶段6 加交互
   → 阶段7 手机+内容 → 阶段8 上线 → 阶段9 迭代
```

- **阶段 0–4 是"想清楚"**（你想、Codex 帮你整理成文档）。这部分**几乎不写业务代码**，但最重要。
- **阶段 5–8 是"做出来"**（Codex 写、你验收）。
- **阶段 9 是"活下去"**（长期维护，主要是你的事）。

## 阶段 0 · 项目初始化

**做什么**
- 在项目目录（`MyWebsite\`）里用 Git 建仓库。
- 建好基本骨架文件：`PRD.md`、`ARCHITECTURE.md`、`PROGRESS.md`、`DECISIONS.md`（四个文件先建空模板，后面阶段往里填）。
- 让 Codex 说明它准备用什么工具链（先不动手，先汇报方案）。

**为什么**：地基。没有 Git 就没有"后悔药"；没有四个文档文件，决策就只在聊天记录里，迟早乱。

**交付物**
- 一个 Git 仓库（首次提交完成）
- 四个空的文档骨架文件

**验收标准**
- [ ] Codex 能告诉你"项目当前在 Git 的第几个版本"
- [ ] 四个文件都在项目根目录能看到

> 📌 话术见《沟通手册》第 4 节。

---

## 阶段 1 · 写产品需求文档（PRD）

**做什么**：你口述 / 填空，Codex 帮你整理成 `PRD.md`。这是整个项目的"说明书"。

**为什么**：PRD 是后续所有决策的依据。Codex 每做一件事，都该回头对照 PRD——这就是"架构清晰"的源头。

**填空模板**（你填，Codex 润色）：

```
# 我的个人网站 PRD

## 1. 我是谁
（一句话：你的身份、你正在做什么）

## 2. 网站写给谁看
（主要访客是谁？例如：我的学员 + AI 初学者）

## 3. 网站定位
课程/教学站：用章节式结构展示我的 AI 学习内容，引导学员系统跟学。

## 4. 核心页面（每个页面写一句话用途）
- 首页：第一眼让人知道我是谁、这里有什么
- 课程目录页：所有学习内容的章节列表（带进度）
- 单篇内容页：一篇学习笔记/课程的正文（带评论区）
- 关于我：我的介绍 + 联系方式
- （可选）订阅页 / 资源链接页

## 5. 想要的交互
- 评论（读者能留言）
- 邮件订阅（读者能订阅更新）
- 课程进度（学员能看到自己学到哪了）
- 不要：用户登录系统、数据库、付费（暂时）

## 6. 我希望的风格感受
（等阶段3看完 demo 再填具体方向；现在先写感受词，如"专业、好懂、不花哨"）

## 7. 非目标（明确不做什么，防止需求膨胀）
- 不做论坛、不做社交
- 第一版不上付费/会员
```

**交付物**：填好的 `PRD.md`（Codex 确认它完全理解了）

**验收标准**
- [ ] Codex 能用自己的话复述你的 PRD，你听了觉得"对，就是这个意思"
- [ ] PRD 里写清楚了"非目标"（不做什么）

## 阶段 2 · 技术选型

**做什么**：让 Codex 推荐 2–3 个技术方案，对比优劣，你拍板，记进 `DECISIONS.md`。

**为什么**：技术栈是地基中的地基。非程序员不需要懂代码细节，但要懂"选它的理由"和"换它的代价"。

**你要懂的概念**（科普）：
- **静态网站（推荐你选这个）**：网站就是一堆预先写好的页面，像一本印好的书。便宜、快、安全、手机友好。你的"课程/教学站 + 轻交互"完全够用。
- **动态网站/全栈**：有服务器、数据库，能做登录、付费。复杂贵，你暂时不需要。

**倾向方案**（写进 PRD 让 Codex 评估，最终 Codex 推荐你定）：

| 层 | 倾向方案 | 备选 | 为什么 |
|----|----------|------|--------|
| 框架 | **Astro** | Docusaurus | Astro 内容驱动、性能好、个性化空间大；Docusaurus 最快得到带侧边栏目录的课程站骨架 |
| 内容格式 | Markdown/MDX | — | 你能像写文档一样写文章，不用学编程 |
| 评论 | Giscus | Utterances | 免费，基于 GitHub Discussions |
| 邮件订阅 | Buttondown | — | 免费邮件订阅服务 |
| 课程进度 | localStorage | — | 存在访客浏览器里，不需要后端 |
| 部署 | Vercel | Netlify | 免费、一键上线、手机适配好 |

**交付物**：`DECISIONS.md` 里一条带理由的决策记录

**验收标准**
- [ ] Codex 给了你至少 2 个方案的对比表（含"换它的代价"）
- [ ] 你能用自己的话说出"为什么选这个"
- [ ] 决策写进了 `DECISIONS.md`

> 📌 话术见《沟通手册》第 6 节。

## 阶段 3 · 找设计参考（demo）

**做什么**：让 Codex 推荐一批优秀的个人网站/课程站 demo，你浏览、挑出喜欢的，把链接/截图存进 `DECISIONS.md`。

**为什么**："个性化"不是靠形容词（"高级感""简约"）能传达到 Codex 的——它听不懂。**靠参考站最准**。你看真实例子，选出你喜欢的，Codex 照着做。

**怎么"看 demo"——5 个观察点**（每个 demo 都按这 5 点打分）：

| 观察点 | 你要问自己 |
|--------|-----------|
| ① 布局 | 它的内容是怎么排的？是左右分栏、还是上下流？我喜欢吗？ |
| ② 配色 | 主色是什么？深色还是浅色？给人什么感觉？ |
| ③ 字体 | 标题字是粗是细？是衬线（有装饰）还是无衬线（干净）？ |
| ④ 导航 | 顶部菜单？侧边栏目录？找东西方不方便？ |
| ⑤ 手机效果 | 缩小到手机宽度还好看吗？（用浏览器开发者工具模拟，或直接手机打开） |

**风格决策表**（你看完 demo 后填，给 Codex 当依据）：

```
我喜欢的参考站：
- [链接1] —— 喜欢它的：______（如侧边栏目录）
- [链接2] —— 喜欢它的：______（如配色）
不要的元素：______（如"不要大图轮播""不要弹窗"）
综合方向：______（如"参考1的布局 + 参考2的配色，整体偏极简"）
```

**交付物**：`DECISIONS.md` 里一份"设计参考 + 风格方向"

**验收标准**
- [ ] 你至少看过 5 个 demo
- [ ] 选出了 2–3 个明确喜欢的，并写明"喜欢它的什么"
- [ ] 写出了"不要什么"

> 📌 话术见《沟通手册》第 5 节。

## 阶段 4 · 信息架构设计

**做什么**：让 Codex 根据你的 PRD + 选定的风格，起草网站的**地图、导航、文件夹结构**，写进 `ARCHITECTURE.md`。

**为什么**：架构清晰 = 以后加内容、改东西不乱。这一步定好"骨架"，后面填内容才顺手。

**课程/教学站典型结构**（Codex 会按你的实际内容调整）：

```
网站地图：
首页
├── 课程目录（按章节/主题组织）
│   ├── 第一章：xxx
│   │   ├── 1.1 第一节
│   │   └── 1.2 第二节
│   └── 第二章：xxx
├── 关于我
└── 订阅

对应的文件夹结构（示例）：
MyWebsite\
├── src\
│   ├── pages\          ← 页面
│   │   ├── index.md        （首页）
│   │   ├── about.md        （关于我）
│   │   └── courses\        （课程目录和内容）
│   │       ├── index.md        （课程总目录）
│   │       ├── 01-第一章\
│   │       │   ├── 1-1-第一节.md
│   │       │   └── 1-2-第二节.md
│   │       └── 02-第二章\
│   ├── components\     ← 可复用的小部件（导航、评论框等）
│   └── layouts\        ← 页面框架（顶栏、页脚）
├── public\             ← 图片等静态资源
└── docs\               ← 你现在看的文档
```

**交付物**：`ARCHITECTURE.md`（含网站地图 + 目录结构 + 导航方案）

**验收标准**
- [ ] 你看着这张地图，能指出"我的每一类内容将来放在哪"
- [ ] Codex 解释了"为什么这样分"（例如：按章节分文件夹，方便以后加新内容不乱）
- [ ] 你确认后，Codex 才开始写代码

## 阶段 5 · 分页面实现（核心施工）

**做什么**：**一次只做一个页面**，做完、验收、Git 提交，再做下一个。顺序建议：

1. **首页** → 2. **课程目录页** → 3. **单篇内容页** → 4. **关于我**

**为什么"一次一个"**：这是工程化最重要的纪律。一次性做完所有页面 = 回到业余模式，改不动、查不出问题。

**每个页面的标准流程**：
```
你下任务（用《沟通手册》话术）
  → Codex 做这个页面
  → 你本地预览（手机+电脑都看）
  → 不满意？指出具体哪里（"标题再大点""这里的颜色太深"）
  → 满意 → Codex 提交 Git 版本
  → 下一个页面
```

**交付物**：每个页面一个 Git 提交；最终一个能点击浏览的雏形网站

**验收标准**（每个页面都要过）
- [ ] 这个页面在电脑上显示正常
- [ ] 这个页面在手机宽度下显示正常（**重点**）
- [ ] 页面之间能点链接互相跳转
- [ ] 内容用的是你的真实内容（哪怕先放一篇），不是 Lorem Ipsum 占位文

> 📌 话术见《沟通手册》第 7 节。

## 阶段 6 · 加轻交互

**做什么**：在已经做好的页面上，加上：
- **评论**（Giscus）——读者能留言
- **邮件订阅**（Buttondown）——读者留邮箱接收更新
- **课程进度**（localStorage）——学员能看到"已学/未学"

**为什么放最后加**：先把内容骨架做扎实，再加花活。顺序反了会被交互 bug 拖累。

**交付物**：能评论、能订阅、能记进度的网站

**验收标准**
- [ ] 你自己发了一条评论，能看到
- [ ] 你输入邮箱订阅，能收到确认邮件
- [ ] 标记一篇内容"已学"，刷新页面进度还在

---

## 阶段 7 · 手机适配 + 内容填充

**做什么**
- **手机适配打磨**：用手机实际打开每个页面，找出歪了/挤了/字太小的地方，让 Codex 调。
- **内容填充**：把你的真实学习笔记/课程内容，一篇篇填进去（这部分主要是**你**写）。

**为什么单独成阶段**：手机体验是你的核心诉求之一，必须专门过一遍；内容是网站的灵魂，不能一直放占位文。

**交付物**：手机端体验良好、内容充实的网站

**验收标准**
- [ ] 用手机打开网站，每个页面都好读、不歪
- [ ] 至少填了 3–5 篇真实内容
- [ ] 图片在手机上不会撑爆屏幕

---

## 阶段 8 · 部署上线

**做什么**：把网站部署到 **Vercel**（免费），拿到一个公开网址。

**为什么**：本地预览只有你自己看得到，部署后全世界都能访问。

**大致步骤**（Codex 会手把手带你点）：
1. 把项目推到 GitHub（一个放代码的网站）
2. 在 Vercel 用 GitHub 账号登录
3. 导入项目 → 一键部署
4. 拿到网址，手机打开验证

**交付物**：一个公开可访问的网址（如 `your-site.vercel.app`）

**验收标准**
- [ ] 用手机浏览器打开网址，网站正常
- [ ] 发给一个朋友，他也能打开
- [ ] （可选）绑定你自己的域名

---

## 阶段 9 · 持续迭代（长期）

**做什么**：网站上线只是开始。日常你要：
- **写新内容** → 放进对应文件夹 → 提交 → Vercel 自动更新
- **小优化** → 一次改一个地方，改完提交
- **备份** → 代码在 GitHub 上就是备份；重要内容自己也留一份

**核心纪律（终身适用）**
- 每次改动都先 Git 提交当前状态（"拍快照"），改坏了能回退
- 一次只改一件事，改完验证再改下一件
- 大改动前，让 Codex 先说"我打算怎么改"，你确认后再动

## 13. 每阶段验收标准速查表

| 阶段 | 最关键的 1 条验收 |
|------|------------------|
| 0 初始化 | Git 有首个提交，四个文档文件在 |
| 1 PRD | Codex 能复述 PRD 且你认可 |
| 2 选型 | 决策写进 `DECISIONS.md`，你能说出"为什么选它" |
| 3 参考 | 选定 2–3 个参考站并写明喜欢什么 |
| 4 架构 | 你能指着地图说出每类内容放哪 |
| 5 分页面 | 每个页面手机+电脑都正常，用真实内容 |
| 6 交互 | 评论/订阅/进度三个功能都能用 |
| 7 手机+内容 | 手机体验顺、至少 3–5 篇真实内容 |
| 8 上线 | 公开网址手机可访问，朋友也能打开 |
| 9 迭代 | 养成"每次改动先 Git 提交"的习惯 |
