# Codex 沟通手册：与 AI 编程助手合作的标准话术

> 这份手册解决一个问题：**不懂编程的你，怎么把需求说清楚，让 Codex 做出你想要的东西。**
> 里面全是**可直接复制粘贴的话术**。每个话术都对应《开发路线图》里的一个阶段。
> 配合 [`01-开发路线图.md`](./01-开发路线图.md) 使用。

---

## 0. 怎么用这份手册

- 每次跟 Codex 对话前，**先翻到对应阶段的话术**，复制 → 改掉方括号 `[...]` 里的内容 → 发送。
- 话术里的方括号 `[...]` 是**你要替换成自己信息**的地方。
- **不要**自由发挥改话术的结构——这些结构是经过设计的（见第 3 节"标准沟通模板"）。
- 第一件事：把这三份文档都丢给 Codex 让它读过，你们就用同一套语言了。

---

## 1. 和 AI 编程工具沟通的 5 条黄金法则

| # | 法则 | 为什么 | 怎么做 |
|---|------|--------|--------|
| 1 | **给全背景** | Codex 不认识你，背景越全它越准 | 每次先说清"我是谁、要什么、约束是什么" |
| 2 | **一次只做一件事** | 一次塞太多，它会顾此失彼 | 一次只让它做一个页面/一个功能 |
| 3 | **用文件存上下文** | 聊天记录会丢，文件不会 | 决策、架构、进度都写进项目里的 `.md` 文件 |
| 4 | **要可验证的产出** | "能跑起来看到"才算数 | 每次都要能在浏览器里看到结果 |
| 5 | **出错贴原文** | 你转述报错会丢关键信息 | 报错信息**整段复制**给 Codex |

## 2. 用文件管理上下文（项目的 4 个核心文档）

这 4 个文件放在项目根目录，是项目的"长期记忆"。Codex 每次做事都该读它们：

| 文件 | 作用 | 什么时候更新 |
|------|------|-------------|
| **`PRD.md`** | 产品需求：你是谁、写给谁、有什么页面 | 阶段 1 写，之后偶尔补充 |
| **`ARCHITECTURE.md`** | 信息架构：网站地图、目录结构、导航 | 阶段 4 定，之后基本不变 |
| **`DECISIONS.md`** | 决策记录：技术选型、设计风格的结论 + 理由 | 每次做关键决定时追加一条 |
| **`PROGRESS.md`** | 进度日志：做到第几阶段、每次做了啥 | 每个阶段结束时追加 |

> 💡 给 Codex 的开场白里永远带一句："请先读项目里的 PRD/ARCHITECTURE/DECISIONS/PROGRESS。"

---

## 3. 标准沟通模板（每次对话的骨架）

跟 Codex 说任何一件事，都用这个 5 段式结构：

```
【背景】我是谁、项目是什么、当前在哪一步
【目标】这次我想达成什么（一次只说一件事）
【约束】有什么限制/偏好（如"不懂编程""手机要好看""不要花哨"）
【交付物】我要拿到什么具体的东西
【验证方式】我打算怎么检查你做得对不对
```

> 下面所有话术，都是这个模板的具体填法。

## 4. 第 1 次启动 Codex（阶段 0 初始化）— 完整话术

**什么时候用**：项目刚开始，第一次正式跟 Codex 合作。

```
你好 Codex，我要做一个个人网站。请先读一下项目目录 docs/ 里的三份文档：
《00-启动清单》《01-开发路线图》《02-Codex沟通手册》。
读完请用一句话告诉我你理解了我的工作方式。

【背景】
- 我完全不懂编程，这是我的第一个网站项目。
- 网站定位：课程/教学站（章节式结构 + 学习路径，引导学员跟学我的 AI 学习内容）。
- 交互需求：轻交互（内容 + 评论 + 邮件订阅 + 课程进度，不上数据库/后端）。
- 设计风格：未定，计划在阶段3看你推荐的 demo 后再定。

【目标】现在执行《开发路线图》的【阶段 0：项目初始化】。

【约束】
- 我是 Windows 系统。
- 先不要写业务代码。
- 每一步都先告诉我你打算怎么做，等我确认再动手。

【交付物】
- 一个初始化好的 Git 仓库（含首次提交）
- 四个空的文档骨架：PRD.md、ARCHITECTURE.md、DECISIONS.md、PROGRESS.md

【验证方式】
做完后告诉我：当前 Git 在第几个版本、四个文件是否就位。
```

## 5. 让 Codex 推荐设计参考（阶段 3）— 完整话术

**什么时候用**：阶段 2 选型完成后，你不知道想要什么风格，要 Codex 给参考。

```
【背景】项目已完成阶段0-2，现在进入【阶段3：找设计参考】。请先读 PRD.md 和 DECISIONS.md。

【目标】给我推荐一批优秀的参考网站/demo，帮我决定设计风格。

【约束】
- 参考类型要匹配"课程/教学站 + 个人 AI 学习分享"。
- 我想看的风格尽量多样：极简技术感、温暖人文、现代卡片式、手写笔记感，各给几个。
- 每个参考请附：网址 + 一句话说明它好在哪里。
- 优先给开源/免费可参考的、手机端也做得好的。

【交付物】
一份参考清单，按风格分类，每类 3-5 个，含网址和亮点说明。

【验证方式】
我会逐个用手机和电脑打开看，用《路线图》第3节的"5个观察点"打分，然后告诉你我选哪几个。
```

**你反馈选择时，用这个格式**（看完 demo 后发给 Codex）：

```
我看完了，我的选择：
- 喜欢 [参考站A链接]：喜欢它的 [侧边栏目录 + 深色配色]
- 喜欢 [参考站B链接]：喜欢它的 [大标题字体 + 留白]
- 不喜欢 [参考站C]：因为 [轮播图太花哨]
综合方向：参考A的布局 + 参考B的字体，整体偏极简，深色为主。
请把这个决定记进 DECISIONS.md。
```

> ⚠️ 关键：**用具体的参考站+具体元素描述，永远比"高级感/简约"这种形容词准。** 详见第 10 节。

## 6. 让 Codex 推荐技术栈（阶段 2）— 完整话术

**什么时候用**：PRD 写完后，要定用什么技术。

```
【背景】PRD 已完成（请先读 PRD.md）。现在进入【阶段2：技术选型】。

【目标】给我推荐 2-3 个适合我需求的技术方案，对比优劣，让我选。

【约束】
- 我不懂编程，方案要能让我以后自己用 Markdown 写内容、不用写代码。
- 倾向"静态网站"路线（轻交互、免费托管、手机友好），但请你也评估是否真的合适。
- PRD 里写了倾向方案（Astro + Giscus + Buttondown + Vercel），请评估是否合理，或给更好的。

【交付物】一张对比表，含：方案名 / 优点 / 缺点 / 换它的代价 / 是否推荐。并给出你的最终推荐和理由。

【验证方式】
我会问你"如果将来我想加 XX 功能，这个方案还能撑住吗"，你能回答即可。
确定后请记进 DECISIONS.md。
```

**决策表**（Codex 给你后，你照着填确认）：

```
最终选定方案：__________
理由（用我的话说）：__________
将来加功能的代价：__________
确认记入 DECISIONS.md：是
```

## 7. 分阶段下任务（阶段 1、4、5、6）— Prompt 模板

### 7.1 阶段 1：写 PRD

```
【背景】阶段0已完成。现在【阶段1：写PRD】。
【目标】我口述需求，你帮我整理成规范的 PRD.md。
【约束】用《路线图》阶段1的填空模板。我不懂技术，请用大白话。
【交付物】一份填好的 PRD.md，含"非目标"（明确不做什么）。
【验证方式】你用你自己的话复述一遍 PRD，我确认"对，就是这个意思"。
```

### 7.2 阶段 4：设计信息架构

```
【背景】PRD和技术选型已定（请读 PRD.md、DECISIONS.md）。现在【阶段4：信息架构】。
【目标】起草网站地图、导航结构、项目目录结构，写进 ARCHITECTURE.md。
【约束】按"课程/教学站"的章节式结构设计，方便我以后加新内容不乱。
【交付物】ARCHITECTURE.md（含地图 + 目录结构 + 导航方案）。
【验证方式】给我解释"为什么这样分"，我能看懂并确认后再进入下一步。
```

### 7.3 阶段 5：做一个页面（每个页面单独发一次）

```
【背景】请读 PRD.md、ARCHITECTURE.md、DECISIONS.md。现在【阶段5：分页面实现】。
本次只做：[首页 / 课程目录页 / 单篇内容页 / 关于我 —— 选一个]。

【目标】把这个页面做出来，能本地预览看到。

【约束】
- 严格遵循已定的设计风格（见 DECISIONS.md 的参考站方向）。
- 必须手机适配（做好后我会用手机看）。
- 内容用我的真实内容（我先给你一段），不要用占位假文。
- 不要顺手改其他页面。

【交付物】这个页面能跑、能在浏览器打开。

【验证方式】
- 你告诉我怎么本地预览（给具体命令，一步步教）。
- 我会在电脑和手机上都看一遍，再告诉你哪里要调。
- 调到满意后，请你做一个 Git 提交，提交信息写"完成 XX 页面"。
```

### 7.4 阶段 6：加一个交互功能（每个功能单独发一次）

```
【背景】现在【阶段6：加轻交互】。本次只加：[评论 / 邮件订阅 / 课程进度 —— 选一个]。

【目标】实现这个功能，能在已做好的页面上用。

【约束】
- 用免费、不需要后端的方案（PRD 里倾向：评论用Giscus、订阅用Buttondown、进度用localStorage）。
- 一步一步教我配置（有些需要我去第三方网站注册拿 key）。

【交付物】这个功能能用。

【验证方式】我自己实际操作一遍（发评论/订阅/标记进度），告诉你结果。
通过后做一个 Git 提交。
```

## 8. 验收清单：怎么检查 Codex 的产出

每次 Codex 说"做完了"，别急着说好。先按这个清单查：

**本地预览**（让 Codex 教你启动）
- [ ] 电脑浏览器打开，页面正常显示
- [ ] 点页面里的链接，能跳到正确的地方
- [ ] 没有报错（让 Codex 帮你看控制台有没有红字）

**手机预览**（必做）
- [ ] 把浏览器窗口拖窄到手机宽度，页面不歪、不溢出
- [ ] 或：让 Codex 教你怎么用手机打开本地预览
- [ ] 字在手机上能看清，按钮能点

**功能自测**
- [ ] 按"交付物"逐条核对，每一条都真的去试一遍

**通过后**
- [ ] 让 Codex 做一次 Git 提交，提交信息清楚说明做了什么
- [ ] 让 Codex 在 PROGRESS.md 追加一行进度

## 9. 出错时怎么沟通（重要）

**铁律：不要慌、不要乱改、不要自己瞎删文件。**

### 报错时的标准话术

```
出问题了。我做了 [描述你的操作，例如"我点了预览按钮"]，然后出现了下面这个错误。
请【整段报错信息】见下：

[把报错信息完整粘贴在这里，一个字都不要改]

请告诉我：
1. 这个错误是什么原因（用大白话）？
2. 你打算怎么修？
我先不急，你解释完方案、我确认后再动手修。
```

### 改不好怎么办

```
这个问题改了几次还没好。先停下，不要再改了。
请帮我把项目 Git 回退到 [上一个能用的版本 / 刚才那个提交]。
回退后我们重新想方案。
```

> 💡 这就是 Git 的价值——**永远能回到能用的版本**。所以前面每步都要提交。

## 10. 把"个性"翻译成 Codex 能执行的需求

业余说法（Codex 听不懂）→ 工程化说法（Codex 能照做）：

| ❌ 别这么说 | ✅ 这样说 |
|------------|----------|
| "要有高级感" | "参考 [网址] 的排版，标题用无衬线字体，大量留白，主色用深灰" |
| "要简约" | "去掉 [参考站C] 里的轮播图和侧边广告位，只保留导航+正文+页脚" |
| "要活泼一点" | "像 [网址] 那样，标题加一个手写体强调色，配几张插图" |
| "颜色好看点" | "主色改成 #2D6A4F（墨绿），背景用 #F8F9FA（米白）" |
| "感觉不对" | "具体不对在：[标题太小 / 行距太挤 / 手机上按钮挤在一起]" |

**通用法则**：
1. **给参考**：一个参考站胜过一万句形容词。
2. **指具体**：说"标题""按钮""侧边栏"，不要说"整体感觉"。
3. **改一处说一处**：一次只反馈一个问题，改完再看下一个。
4. **配截图**：哪里不对，截图圈出来发给 Codex，最直观。

## 11. Git 版本管理配合话术

**每个阶段/每个页面做完后，固定说这句**：

```
这一步我验收通过了。请：
1. 做一次 Git 提交，提交信息写：[完成 XX]
2. 在 PROGRESS.md 追加一行：[日期] - [完成了什么]
3. 告诉我现在项目在第几个版本。
```

**大改动前**：

```
接下来我想改 [XX]，这个改动比较大。
请先告诉我你打算怎么改、会影响哪些地方，我确认后你再动手。改之前请先提交一个当前状态的版本。
```

## 12. 常见坑与避坑指南

| 坑 | 后果 | 怎么避 |
|----|------|--------|
| 一次让 Codex 做完整个网站 | 改不动、查不出问题 | **严格一次一个页面/一个功能** |
| 频繁推翻重来 | 时间全浪费在返工 | PRD 阶段想清楚；决策记进文件，不轻易推翻 |
| 不勤提交 Git | 改坏了回不去 | 每个阶段、每个验收通过都提交 |
| 自己乱改不懂的配置文件 | 项目跑不起来 | 不懂的文件**只读不写**，改什么让 Codex 来 |
| 用形容词描述需求 | Codex 做出来不是你要的 | 用参考站 + 具体元素（第 10 节） |
| 报错自己转述 | 丢关键信息 | **报错原文整段粘贴** |
| 让 Codex 直接改生产环境 | 上线的站可能坏 | 大改动先在本地做、验证、再部署 |
| 忘记更新 PROGRESS.md | 不知道做到哪了 | 每次提交时顺手追加一行 |

## 🎯 一句话总结

> **你想清楚"要什么"（用 PRD 和参考站），Codex 负责"怎么写"，文档和 Git 把过程钉死。**
> 遇到任何卡壳，回到这句话，回到当前阶段的话术，一步一步来。
